## ある朝、Google Cloudから通知が来た

「お支払いが確定しました」

金額を見て固まりました。**¥12,800**。

自分が作っているのは、AmazonレビューのAI判定をするChrome拡張「Sakura Score」です。まだChrome Web Storeの審査中で、使っているのは自分だけ。**ユーザー0人**。

しかもこれが初めてではありません。その少し前に、AIエージェントツール「OpenClaw」の設定を試していたときにも、Gemini ProのAPIが裏で動いていることに気づかず放置。何の成果もないまま**¥5,000**の請求が来ていました。

合計**約¥18,000**。1ヶ月で消えました。

個人開発者にとっての¥18,000は、ドメイン数年分、サーバー数ヶ月分です。それが「ユーザー0人」のアプリと「設定しただけ」のツールに消えた。正直へこみました。

## 何を作っていたのか

Sakura Scoreは、Amazonの商品レビューをAIで分析して「サクラ度」を判定するChrome拡張です。

もともとレビュー分析にはOllama（ローカルで動くLLM）を使っていました。自分のPCで動くので、何万回呼んでも電気代だけ。無料です。

ところがOllamaだと処理が遅く、Chrome拡張としてはレスポンスが悪い。そこでGoogle Gemini API（gemini-2.5-flash）に乗り換えました。

乗り換えるとき、AIに確認しました。「無料枠の範囲でやりたい」「課金されない設定になっているか」。AIは「大丈夫です」と答えてくれた。指示通りにやったつもりでした。

今思えば、AIのすごさに舞い上がっていたのかもしれません。コードを書いてくれる、設計もしてくれる、質問には何でも答えてくれる。そんなAIが「無料で大丈夫」と言っているのだから大丈夫だろう、と。

## 事故の規模

Google Cloudのコンソールを開いて、過去30日の数字を見たときは目を疑いました。

- リクエスト数: **約2,000回**
- 出力トークン: **3,174万トークン**
- 1日あたりの呼出し: **約561回**
- ユーザー数: **0人**

誰も使っていないアプリのクローラーが、毎日561回もAPIを叩いていたんです。

## なぜ気づかなかったのか ― 4つの盲点

### 盲点1: 「ローカルLLMの感覚」が抜けなかった

Ollamaは自分のPC上で動くLLMです。何回呼んでも0円。APIキーもクラウドも関係ありません。

Sakura ScoreのクローラーはOllama時代に作ったもので、asins.json（1,152件のAmazon商品）を30〜60秒間隔で巡回し、1つずつAIにレビュー分析を投げる設計でした。Ollamaなら何の問題もありません。

このクローラーのAPI呼出し先をOllamaからGemini APIに差し替えたとき、**呼出し量はそのまま**にしました。「動いてるし大丈夫だろう」。それが1日561回、月に約2,000回のAPI呼出しになっていたわけです。

**ローカルLLMの「タダで回し放題」の感覚は、クラウドAPIに持ち込むと事故になります。**

### 盲点2: AIに「無料か」を確認しても防げない

これが一番伝えたいことです。

OpenClawでもSakura Scoreでも、AIに何度も確認しました。「これ無料の範囲でできる？」「課金されない？」「この設定で大丈夫？」。そのたびに「大丈夫です」「無料枠内で収まります」と返ってきました。

指示通りにやりました。でも課金されました。

**AIは嘘をついたわけではありません。** でも「無料枠内です」という回答は、そのときの設定やトークン量が本当に無料枠に収まるかを保証するものではなかった。AIは料金表の知識を持っているだけで、僕のGCPコンソールの実際の状態は見えていません。

AIはすごいです。コードも書けるし、設計もできる。でも、すごいからこそ信じすぎた。「AIがOKと言っている＝本当にOK」だと思い込んでいました。

そして厄介なのは、**有料課金されていると知っていればクレカの対策もできた**ということです。「無料でやれている」と信じていたから、防衛策を取る発想すらなかった。

### 盲点3: レート制限は無料の意味ではない

Google Cloudのコンソールには「RPD（Requests Per Day）」という数字が表示されます。1日あたりのリクエスト上限です。

これを見たとき、「制限があるから使いすぎない仕組みになっているんだな」と安心しました。

**全然違いました。**

RPDはリクエスト数の制限であって、課金とは無関係です。実際の課金はトークン量ベース。つまり、リクエスト数が制限内でも、1回あたりのトークン消費が多ければ料金は膨らみます。

Gemini 2.5 Flashは「安い」と言われますが、3,174万トークンを流せば当然¥12,800にもなる。料金表を見たときに「Flash = 安い」というイメージで思考停止して、実際の使用量で計算しなかったのが失敗でした。

### 盲点4: API呼出し経路が3つあった

Sakura Scoreのアーキテクチャには、Gemini APIを呼ぶ経路が3つありました。

1. **agent.js**（ローカルのクローラー）→ 直接Gemini APIを呼出し
2. **Supabase Edge Function** → Gemini APIを呼出し
3. **Cloudflare Worker** → Gemini APIを呼出し

agent.jsのGeminiフラグをfalseにしても、Edge FunctionやCloudflare Workerは動き続けます。1つ止めたつもりでも、他の経路からAPIが叩かれ続ける。

**自分のプロジェクトからAPIを呼んでいる場所が全部でいくつあるか、即答できますか？** 即答できないなら、それは事故の種です。

## 発覚から対策完了まで

| 経過 | やったこと |
|------|-----------|
| 0分 | GCPコンソールで請求額を確認。絶望する |
| 15分 | agent.jsの`USE_GEMINI`フラグを`false`に変更 |
| 30分 | GCPコンソールでGemini APIそのものを無効化 |

| 90分 | GCPの請求先カードを**残高0のプリペイドカード**に変更 |
| 120分 | 対策完了 |

## 完全防衛マニュアル

### 防衛1: プリペイドカード戦略（これだけでいい）

すべての対策の中で、**これが唯一の「確実に機能する」防衛策**です。

1. 残高0のプリペイドカード（Visaプリペイドなど）を用意する
2. GCPの請求先をそのカードに変更する
3. APIを使いたいときだけ、必要な分だけチャージする

これで、設定をどれだけミスしても、APIが暴走しても、課金は物理的に止まります。残高がなければ請求は通りません。

**重要:** YouTube PremiumなどGCP以外の定額課金は別のカードにしておくこと。GCPの請求先だけを変えれば、他のGoogleサービスには影響しません。

なぜこれが最強かというと、他の対策はすべて「正しく設定できていれば」という前提があるからです。

- アラート設定 → 設定場所がわからない、設定したつもりでミスしている可能性
- 予算上限 → 上限を超えてもAPIが即座に止まるとは限らない
- APIキー管理 → 複数経路があると漏れる

プリペイドカードだけは、設定の知識がなくても、慣れないサービスでも、確実に動きます。

### 防衛2: ローカルLLM → クラウドAPI 移行チェックリスト

- **現在の呼出し回数を計測する。** ログに出力して1日の呼出し回数を数える。「たぶんこのくらい」ではなく実数を出す
- **その回数でAPIを回したらいくらか計算する。** 料金表の「1Mトークンあたり$X」にレビュー1件あたりの平均トークン数を掛ける
- **最初の1週間は呼出し量を1/10にする。** いきなり本番の量を流さない
- **月額上限アラートを設定する。** GCPなら「予算とアラート」から設定できる

### 防衛3: API経路の棚卸し

- ソースコード全体で`fetch`や`axios`でAPIのURLを叩いている箇所を検索する
- 各経路にキャッシュがあるか確認する
- 各経路にkill switchがあるか確認する（フラグ1つで止められるか）
- 「誰も使っていないのに回っている処理」がないか確認する

## ¥18,000で買った教訓

アラート設定、予算上限、API無効化、キー管理、使用量モニタリング……。正しい対策はたくさんあります。AIに聞けば全部教えてくれます。でも、**慣れないサービスでそれを完璧にやるのは無理です。**

AIに「無料か」を聞いて安心していた時間が、そのままプリペイドカードを設定する時間に使えていれば、1円も失わなかった。

**結局、最初の5分でプリペイドカードを1枚設定するだけで、¥18,000は全額防げました。**

難しい対策を10個やるより、簡単な対策を1つ確実にやる方が強い。

これが¥18,000で買った、一番大事な教訓です。
