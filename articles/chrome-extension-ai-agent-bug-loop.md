## はじめに

「Amazon でついつい Prime に誘導される」「カートページの広告がうるさい」——そんな不満を解消するChrome拡張を作りました。

機能はこんな感じです。

- Prime 誘導バナー・お急ぎ便の非表示
- カートページの広告ブロック
- Prime 無料体験の自動スキップ
- サブスク・保証プランへの警告表示
- 「注文を確定する」ボタンを誤タップしにくい位置へ移動

全部で 10 機能ほど。自分用に作り始めたものです。

そしてこの拡張、コードを書いたのはほぼ **Claude（claude-sonnet-4-6）と Codex** でした。私はほぼディレクター役です。

**結論を先に言うと：AIに最初から「完了の定義」を明文化しておかなかったせいで、同じバグを何度も修正する羽目になりました。**

---

## 開発体制：2 つの AI を使い分ける

Claude Code（Claude の CLI）と Codex を並行して使いました。

- **Claude**：設計・実装・デバッグを担当。会話形式でやり取りしながら進める
- **Codex**：独立したタスクの実装や、Claude が詰まった部分の二次確認に使う

問題は、**2 つの AI が同じリポジトリで作業する**ので、「前の AI が途中まで作業していた」状態が発生することです。

なので最終的に `AI.md` という共通ルールファイルを作りました。

```
# 作業開始時に必ず確認すること
1. git status --short を確認する
2. 既存差分を勝手に戻さない
3. docs/agent-handoff.md を読む
4. 変更済みファイルと未追跡ファイルを見て、前の作業の意図を推測する
```

「前の AI が途中まで作業していた可能性を前提にする」というルールを書いておかないと、Codex が Claude の作業を上書きする、あるいはその逆が起きるんです。

## 最初にハマった罠：テストが通るのに動かない

一番最初にぶつかった壁は「**Jest が通るのに Chrome で動かない**」でした。

Prime お急ぎ便を非表示にする機能を実装したとき、テストは全部グリーン。でも実際の Amazon ページを開くと消えていない。

原因はいくつか重なっていました。

### 1. Amazon の DOM は A/B テストで変わる

セレクタを書いても、別の A/B テストバリアントでは全然違う構造になっている。テストで使った JSDOM は静的なスナップショットなので、こういうブレには気づけません。

セレクタ 1 本に依存した瞬間、Amazon の A/B テストに負けます。

### 2. 「一度見えてから消える」flash 問題

`document_end` で要素を消しても、ページ描画の瞬間に一度 Prime バナーが見えてしまう（flash）。ユーザー体験的にアウトです。

解決策は `document_start` で動く CSS ファイルと専用の start スクリプト（`prime-guard-start.js`）を分離すること。manifest の `run_at: "document_start"` で先に CSS を注入して、要素が表示される前に隠しておく設計に変えました。

```javascript
// prime-guard-start.js（document_start で動く）
// ストレージ応答を待つ間、デフォルト ON で先に隠す
// → 設定が OFF なら後から表示を戻す
```

flash を防ぐには「消す」だけじゃなく「最初から見せない」設計が必要、というのを最初から知っていれば余計な遠回りをしなかったと思います。

### 3. Service Worker が止まる

バックグラウンドの `setInterval` でホットリロードを管理していたのですが、Manifest V3 の Service Worker は Chrome の判断でサスペンドされます。サスペンドされると `setInterval` は当然止まります。

`chrome.alarms` API に切り替えることで解決しましたが、**これを最初から知っていれば 1 行で済む話でした**。

```javascript
// NG：Service Worker がサスペンドされると止まる
setInterval(() => { /* hot reload */ }, 5000);

// OK：chrome.alarms はサスペンドをまたいで動く
chrome.alarms.create('hot-reload', { periodInMinutes: 0.1 });
chrome.alarms.onAlarm.addListener((alarm) => {
  if (alarm.name === 'hot-reload') { /* hot reload */ }
});
```

Manifest V3 を使うなら `chrome.alarms` はほぼ必須です。

## 二度目のハマり：AI が「完了」と言った後も壊れていた

最大の問題はここです。

Checkout ページ（注文確認画面）の配送オプション処理を修正したとき、Claude は「テストが通りました、修正完了です」と報告してきます。

でも実際のブラウザで確認すると——

- 「通常配送 無料」まで消えている
- 有料の「お急ぎ便 ¥510」も消えている
- 消したいのは「Prime 会員特典 明日中にお届け」だけなのに

**「テストが通る ≠ 期待通りに動く」**、これが何度も起きました。

なぜかというと、JSDOM の DOM ツリーと実際の Amazon ページの DOM は別物だからです。特に Checkout ページは：

- 非表示になっている Amazon のモーダル要素が DOM に残っている
- 動的に後から挿入される要素がある
- 同じ CSS クラスが「Prime 特典」にも「有料配送」にも使われている

こういう状況を、テストでは再現できません。AI はテストしか見ていないので、当然テストが通れば「完了」と判断します。

---

## 解決策：「完了の定義」を AI に明文化する

結局、`agent-handoff.md` と `AI.md` に**完了条件を具体的に書くことで解決しました**。

```markdown
## 完了条件

修正完了を名乗る前に、最低限これを確認します：

- npm test -- --runInBand 成功
- npm run build 成功
- Chrome 拡張機能の reload
- 対象 Amazon ページ reload
- 実ブラウザで Prime 誘導だけが消え、通常配送・有料配送・注文確認 UI が残ることを確認
```

そして何が「消えていいもの」で何が「残すべきもの」かを表でまとめました。

| 対象 | 方針 |
|---|---|
| Prime 会員特典のお急ぎ便・日時指定便 | 非表示 |
| 通常配送（無料） | **残す** |
| 有料のお急ぎ便（¥510）、最速（¥610） | **残す** |
| 注文確認ボタン・金額・商品名 | **絶対に壊さない** |

これを書いてから、同じ修正を何度もやり直すループがほぼなくなりました。

**最初からこれを書いておけば良かった。**

## もう 1 つのハマり：ホバーするだけでページが飛ぶ

別の問題も起きました。カートページのレコメンドカルーセルで、マウスをホバーするだけで商品ページに飛んでしまうことがありました。

実ブラウザで再現して初めてわかった問題です。テストでは絶対に気づけない。

原因を掘っていくと、カルーセルアイテムが `mouseenter` イベントで `location.href` を書き換えていました。Chrome 拡張の通常コンテキストからはこれを防げないため、`world: "MAIN"` 指定のスクリプトから `Location.prototype` をパッチして対処しました。

```javascript
// manifest.json で world: "MAIN" を指定したスクリプト内
const _assign = Location.prototype.assign.bind(location);
Object.defineProperty(window, 'location', {
  get() { return location; },
  set(value) {
    if (isHoverNavigation(value)) return; // ホバー起因はブロック
    _assign(value);
  }
});
```

「動かない、原因不明」と AI が言ったときは、実ブラウザの console と accessibility tree を直接見た方が早い、とこのとき学びました。

## まとめ：最初から言っておけば良かった一言

振り返ると、ほとんどのロスは「**AI への完了条件の伝え方**」が甘かったことに起因します。

| やってしまったこと | 最初からやるべきだったこと |
|---|---|
| 「テスト通ったら完了」と暗黙的に思っていた | 実ブラウザ確認を完了条件に明記する |
| 2 つの AI が同じファイルを触る | 引き継ぎルール（agent-handoff.md）を初日に作る |
| setInterval でホットリロード | chrome.alarms を最初から使う |
| document_end で要素を消す | flash 防止のため document_start + CSS を設計に組む |
| セレクタを感覚で書く | A/B テストを前提にページ別挙動マトリクスを作る |

AI 駆動開発は速い。でも **「完了」の定義があいまいなまま任せると、AI は自分が知っている範囲でしか完了を判断しません**。実ブラウザの挙動、Amazon の A/B テスト、Service Worker のライフサイクル——これらは AI の推測が届かない領域です。

そこを明文化するのは人間の仕事、というのが今回の教訓です。
